# UI 質素檢查清單

以下清單用於 Portfolio_V2 的前端 UI、文案、版面與狀態提示調整。原則是先確保專業、穩定、可維護，再考慮美化。

## 1. 字體規範

- 中文優先使用 `PingFang TC`、`Noto Sans TC`、`Microsoft JhengHei`。
- 不要用花俏字體作中文主標題。
- 標題字距維持自然，不要過度壓縮。
- 金額、百分比、數量、日期等數字資訊要使用 `tabular-nums`。

## 2. 顏色規範

- 保留溫暖米白底色，但避免過重漸層與玻璃感。
- 以 token 為主，不要在單一元件內自行定義一套顏色。
- 主要語意色只使用以下類型：
  - 正常：`positive`
  - 注意：`caution`
  - 風險：`danger`
  - 中性：`neutral`
- 文字與背景都要先確認對比足夠。

## 3. 金額與百分比格式

- 金額統一為 `HKD 12,345.67`、`USD 12,345.67`、`JPY 12,345`。
- 百分比統一為 `+3.25%`、`-1.08%`。
- 缺失值統一使用 `—` 或 `未有資料`，不要每頁自訂。
- 不要改底層計算，只改顯示格式。

## 4. 狀態標籤規則

- 正常：`已更新`、`已連接`、`快照完整`
- 注意：`待更新`、`價格過舊`、`待覆核`
- 風險：`連接失敗`、`資料缺失`、`更新失敗`
- 中性：`暫無資料`、`尚未生成`
- 同一個狀態在全系統要使用同一套文案與顏色。

## 5. 表格格式規則

- 金額與百分比靠右。
- 名稱、帳戶、類別靠左。
- 負數使用低飽和紅色，正數使用低飽和綠色。
- 行高、欄距與表頭分隔要一致。
- 數字欄位都要維持可掃讀性，避免因為長字串破壞版面。

## 6. 手機版檢查

- 檢查 `360px`、`390px`、`430px`、`768px`、`1024px` 與 desktop。
- 底部導覽不可遮住主要操作。
- 卡片與表格在窄屏下要維持可讀。
- Hero 區在手機版要收斂，避免第一屏只見裝飾。
- Modal 要能完整閱讀與操作。

## 7. 無障礙檢查

- icon-only button 必須有 `aria-label`。
- Modal 必須有明確 title 與關閉方式。
- 表單欄位要有 `label` 或 `aria-label`。
- 目前頁面導覽狀態要有 `aria-current`。
- focus 狀態要清楚可見。
- 狀態不能只靠顏色表達，文字要同步提供。

## 8. 空狀態與錯誤狀態規則

- 空狀態格式固定為：標題、原因、主要行動、次要行動。
- 錯誤提示要寫清楚發生咩事、可能原因、可以點做。
- loading 不可以整頁空白。
- partial data 要明確標示資料未完整。
- stale data 要提醒最後更新時間。

## 9. 報告頁面用語規則

- `投資總覽` 代表 Dashboard。
- `資產管理` 代表 Assets。
- `資金流水` 代表 Funds。
- `分析與報告` 代表 Analysis。
- `資產走勢` 代表 Trends。
- `交易記錄` 代表 Transactions。
- `快照` 一律指每日資產快照。
- `待覆核` 一律指需要人工確認的資料。
- `待更新` 一律指價格或資料未完成更新。
- `資料完整度` 一律指資料覆蓋與可信程度。
- `基準貨幣` 一律指全系統顯示貨幣。

## 10. 每次改 UI 後要跑的測試指令

1. `npm run typecheck`
2. `npm run lint`
3. `npm run build`
4. `npm run preview -- --host 0.0.0.0 --port 4173`
5. 以瀏覽器或 `curl` 檢查主要頁面：
   - `/`
   - `/assets`
   - `/funds`
   - `/analysis`
   - `/trends`
   - `/transactions`

## 備註

- 如果專案暫時沒有 `typecheck` 或 `lint` script，就改用實際可執行的等效指令。
- UI 修改應盡量小步完成，避免一次過重寫整個頁面。
